## 一、开发环境

### 环境要求

- Node.js >= 16
- npm >= 8

### 常用命令

```bash
npm install          # 安装依赖
npm run dev          # 开发模式（监听文件变化，自动构建）
npm run build        # 生产构建（TypeScript 检查 + esbuild 压缩）
npm run version      # 更新 manifest.json 和 versions.json 中的版本号
```

### 构建配置

- **入口**：main.ts
- **输出**：main.js（CJS 格式，项目根目录）
- **外部依赖**：obsidian, @codemirror/*, electron（不打包）
- **打包工具**：esbuild

## 三、编码准则

### 3.1 Karpathy 原则

**先思考再编码**：
- 明确说出假设。不确定就问。
- 存在多种理解时，全部列出 — 不要默默选一种。
- 有更简单的方案就提出来。

**简洁优先**：
- 最少代码解决问题。不做投机性设计。
- 不加没被要求的功能。
- 只用一次的代码不抽象。
- 不为不可能的场景写错误处理。

**外科手术式改动**：
- 只动必须动的。不顺手"改进"周围的代码。
- 遵循现有风格，即使你会写得不同。
- 只清理你自己改动造成的未使用导入/变量/函数。

**目标驱动执行**：
- 实现前先定义成功标准。
- 多步任务列出简要计划，每步附验证方式。
- 修 bug 前先写能复现的测试。

### 3.2 TypeScript 规范

- 使用严格类型，避免 `any`（配置系统除外）
- 接口优先于类型别名
- 函数参数和返回值都要有类型标注
- 使用 `readonly` 保护不可变数据

### 3.3 Obsidian API 规范

- 使用 `app.vault` 操作文件，不要用 Node.js 的 `fs`
- 使用 `app.workspace` 操作视图和标签页
- 使用 `this.register()` 注册事件监听器，插件卸载时自动清理
- 使用 `this.addCommand()` 注册命令

## 四、Git 工作流

### 4.1 分支策略

```
main          ← 稳定发布版本，始终可构建
  └── dev     ← 日常开发主线
       ├── feature/xxx   ← 新功能
       ├── fix/xxx       ← 修复 bug
       └── refactor/xxx  ← 重构
```

### 4.2 常用操作

**开始新功能**：
```bash
git checkout dev
git pull origin dev
git checkout -b feature/功能名
```

**提交代码**：
```bash
git commit -m "feat: 添加工作台面板组件"
git commit -m "fix: 修复面板拖拽偏移问题"
git commit -m "refactor: 拆分 settings 模块"
git commit -m "docs: 更新 README"
```

**完成功能并合并**：
```bash
git checkout dev
git merge feature/功能名
git push origin dev
```

**发布版本**：
```bash
git checkout main
git merge dev
git tag v1.0.0
git push origin main --tags
```

### 4.3 提交类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 添加逾期提醒功能` |
| `fix` | 修复 bug | `fix: 修复热力图时区问题` |
| `refactor` | 重构（不改变功能） | `refactor: 提取 TaskService` |
| `docs` | 文档 | `docs: 更新架构文档` |
| `style` | 格式调整（不影响逻辑） | `style: 统一缩进风格` |
| `chore` | 构建/依赖/工具变更 | `chore: 升级 esbuild 版本` |
| `test` | 测试相关 | `test: 添加 PlanService 单元测试` |

### 4.4 日常节奏

1. 开始工作前: `git pull origin dev`
2. 小步提交，每完成一个逻辑单元就 commit
3. 功能完成后合并回 dev
4. 稳定后合并到 main 并打 tag

## 五、修改记录规范

**重要规则：每次修改代码后，必须更新修改记录文档。**

### 5.1 记录位置

- 变更日志：`CHANGELOG.md`
- 开发指南：`docs/开发指南.md`（本文件）

### 5.2 CHANGELOG 格式

```markdown
## [版本号] - YYYY-MM-DD

### Added
- 新增功能描述

### Changed
- 变更描述

### Fixed
- 修复描述

### Removed
- 移除描述
```

### 5.3 何时记录

1. **修复 bug 后** — 记录问题、原因、修复方案
2. **新增功能后** — 记录功能描述、涉及文件
3. **重构代码后** — 记录重构原因、改动范围
4. **测试完成后** — 记录测试结果、发现的问题

## 六、开发流程

### 6.1 添加新功能

1. **设计** — 在 `docs/设计书.md` 中记录设计方案
2. **实现** — 遵循现有架构（config → services → views）
3. **测试** — 在测试 vault 中验证功能
4. **文档** — 更新 `CHANGELOG.md` 和相关文档
5. **提交** — 按 Git 工作流提交

### 6.2 修复 Bug

1. **复现** — 在测试 vault 中复现问题
2. **定位** — 找到问题代码
3. **修复** — 最小改动修复问题
4. **验证** — 确认修复有效且无副作用
5. **记录** — 更新 `CHANGELOG.md`

### 6.3 重构代码

1. **分析** — 明确重构目标和范围
2. **备份** — 确保可以回退
3. **重构** — 小步重构，每步可构建
4. **验证** — 确认功能不变
5. **记录** — 更新 `CHANGELOG.md` 和架构文档

## 八、Claude Code 指引

### 8.1 项目指令

本项目通过 `CLAUDE.md` 提供项目级指令，包括：
- 构建和开发命令
- 架构说明
- 编码准则
- 修改记录规范

### 8.2 Superpowers 技能

本项目通过 `.claude/settings.json` 局部启用了 superpowers 插件：

| 技能 | 用途 | 触发方式 |
|------|------|----------|
| brainstorming | 头脑风暴，探索设计方案 | `/brainstorming` |
| writing-plans | 编写实施计划 | `/writing-plans` |
| executing-plans | 按计划执行实施 | `/executing-plans` |
| test-driven-development | 测试驱动开发 | `/test-driven-development` |
| systematic-debugging | 系统化调试 | 遇到 bug 时 |
| requesting-code-review | 请求代码审查 | `/requesting-code-review` |
| verification-before-completion | 完成前验证 | 任务完成前 |

### 8.3 使用原则

- 有 1% 可能用到某个技能，就必须调用它
- 技能优先级：流程技能 > 实现技能
- 用户指令优先级最高，技能次之
