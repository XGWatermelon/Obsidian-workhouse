## 一、设计原则

### 1.1 核心原则：插件不篡改用户文档

插件只在自己的「地盘」（工作台文件夹）里工作，不主动修改或创建用户原本的文件夹结构。

| 操作 | 插件的职责 | 用户的职责 |
|------|-----------|-----------|
| **生成文件/文件夹** | 只在 `工作台/` 下生成 | 用户自己的文件夹由用户管理 |
| **读取文件** | 可以读取整个库 | — |
| **移动/重命名文件** | 不做 | 用户自己处理 |
| **创建外部文件夹** | 不做 | 用户自己处理 |

**为什么这样设计**：
1. **尊重用户** — 用户的笔记库是用户自己的，插件不应该随意改动
2. **避免混乱** — 自动创建的文件夹可能和用户已有的结构冲突
3. **清晰边界** — 插件的东西在工作台里，用户的东西在外面
4. **灵活迁移** — 用户可以随时把内容移进/移出工作台

### 1.2 配置驱动

所有可变行为都通过 config.json 控制，避免硬编码。用户可以通过配置：
- 自定义文件夹位置和名称
- 自定义状态列表和类型列表
- 自定义领域和模块
- 自定义工作流按钮
- 自定义字段映射

### 1.3 按需创建

不在插件启动时批量创建所有文件夹，而是在实际使用时按需创建。避免在用户的 vault 中产生不必要的空文件夹。

## 二、用户画像

**主要用户**：SAP ABAP 开发者 / 技术知识管理者

**使用场景**：
- 每天管理学习计划和工作任务
- 积累技术知识（SAP 各模块、AI 技术等）
- 追踪学习进度和领域覆盖
- 管理稍后阅读的资料

**核心需求**：
- 统一的工作台视图，不用在多个文件夹间跳转
- 快速创建事项和计划
- 可视化的学习进度追踪
- 灵活的配置，适配不同的知识管理习惯

## 三、目录结构设计

### 3.1 推荐结构

```
用户的 Obsidian 仓库/
│
├── 工作台/                          ← 插件的地盘（自动生成）
│   ├── 日计划/                     ← 插件创建
│   │   └── 2026-05-29-日计划.md
│   ├── 周计划/                     ← 插件创建
│   │   └── 2026-W22-本周计划.md
│   ├── 事项/                       ← 插件创建
│   │   └── 2026-05-29-学习TypeScript.md
│   ├── 节点/                       ← 插件创建（知识图谱节点）
│   │   └── AI/
│   │       └── MCP协议.md
│   ├── 待整理/                     ← 插件创建（收件箱）
│   ├── 清单/                       ← 插件创建（生成的统计文档）
│   └── 常用入口.md                 ← 插件创建
│
├── （用户自己的其他文件夹）          ← 用户管理，插件不碰
│
└── .obsidian/
```

### 3.2 文件夹职责

| 文件夹 | 性质 | 时间属性 | 说明 |
|--------|------|---------|------|
| 日计划/ | 任务管理 | 有 | 每日计划，按日期组织 |
| 周计划/ | 任务管理 | 有 | 每周计划，按周数组织 |
| 事项/ | 任务管理 | 有 | 长期任务/项目，有 deadline |
| 节点/ | 知识积累 | 无 | 知识图谱节点，永久存在 |
| 待整理/ | 临时存储 | 无 | 收件箱，待归类内容 |
| 清单/ | 生成文档 | 有 | 工作流生成的统计文档 |
| 常用入口.md | 快捷入口 | 无 | 常用文档链接 |

### 3.3 节点设计

**节点的本质**：知识图谱的基础单元

- **永久性知识** — 不像日计划/周计划有时间属性，节点是永久存在的知识
- **Obsidian 图谱视图的基础** — 节点构成 Obsidian 图谱的基本单元
- **Wiki 链接的目标** — `[[节点/MCP协议]]` 这样的引用
- **帮助 AI 理解知识结构** — 通过节点组织知识体系

**默认位置**：`工作台/节点/`

**可选配置**：如果用户希望节点与工作台平级，可以修改配置：
```json
{
  "basic": {
    "folders": {
      "structure": {
        "nodes": { "name": "节点" }
      }
    }
  }
}
```

### 3.4 路径拼接逻辑

```typescript
function getFolderPath(app, folderKey) {
  const folder = config.basic.folders.structure[folderKey];
  // name 包含 "/" → 绝对路径，直接返回
  if (folder.name.includes("/")) return folder.name;
  // 否则拼接 root
  return `${config.basic.folders.root}/${folder.name}`;
}
```

**示例**：
- `{ name: "日计划" }` → `工作台/日计划`
- `{ name: "工作台/节点" }` → `工作台/节点`（绝对路径）
- `{ name: "节点" }` → `工作台/节点`（拼接 root）

### 4.2 标签页设计

#### 行动指北（ActionTab）

**布局**：左右两栏

| 左栏 | 右栏 |
|------|------|
| 今日要事 | 本周计划 |
| 活跃任务 | 学习进度 |
| 常用入口 | 最近修改 |

**交互**：
- 今日要事：点击任务跳转到日计划
- 活跃任务：点击事项跳转到事项文件
- 常用入口：支持添加/删除常用文档链接
- 学习进度：点击可生成学习状态文档

#### 知识管理（KnowledgeTab）

**内容**：
- 事项清单（进行中）
- 事项池（待评估）
- 最近笔记
- 文档概览（按文件夹统计）

#### 学习分析（AnalysisTab）

**内容**：
- 年度活跃度热力图（GitHub 风格）
- 学习状态分布
- 领域覆盖（按领域/模块展示进度）
- 节点图谱

#### 稍后阅读（ReadingTab）

**内容**：
- 待整理文件夹中的文件列表

## 五、数据模型

### 5.1 事项（Topic）

```yaml
---
tags:
  - 事项
type: 任务              # 任务/想法/项目/写作/学习
status: 进行中          # 待评估/进行中/已完成/已放弃
created: 2026-05-29
updated: 2026-05-29
deadline: "2026-06-15"  # 可选
学习状态: 已阅读        # 可选：待阅读/已阅读/已理解/已掌握
domain: AI              # 可选
module: MCP             # 可选
---
```

### 5.2 日计划

```yaml
---
tags:
  - 计划
  - 日计划
type: 日计划
date: 2026-05-29
weekly_plan: "[[工作台/周计划/2026-W22-本周计划.md]]"
---
```

### 5.3 周计划

```yaml
---
tags:
  - 计划
  - 周计划
type: 周计划
week: "2026-W22"
start_date: 2026-05-25
end_date: 2026-05-31
status: 待评估
---
```

### 5.4 状态机

#### 事项状态

```
待评估 → 进行中 → 已完成
                → 已放弃
```

#### 学习状态

```
待阅读 → 已阅读 → 已理解 → 已掌握
```

## 七、待实现功能

### 7.1 逾期功能

**状态**：待实现

**设计**：

逾期分两种类型：

1. **事项逾期** — 事项文件夹中 deadline < 今日且未完成/已放弃的
2. **计划逾期** — 周计划 end_date 过期且有未完成任务、日记日期过期且有未完成任务

**逾期页面结构**：
```
逾期页面
├── 逾期统计卡片
│   ├── 事项逾期：3个
│   ├── 计划逾期：2个
│   └── 总计：5个
├── 事项逾期列表
│   └── 任务名（逾期 N 天）
└── 计划逾期列表
    └── 计划名（逾期 N 天）
```

**操作按钮**：打开、标记完成、延期、全部标记完成

**待讨论**：计划和事项的模糊边界 — 事项是长期任务（有 deadline），计划是短期执行（周计划/日记），两者的关系需要明确。

## 八、配置设计

### 8.1 配置结构

```
config.json
  ├── version: number
  ├── basic
  │   ├── app: { name, icon, viewTitle }
  │   ├── ui: { openInMainEditor, themeId }
  │   ├── folders: { root, structure }
  │   ├── domains: DomainConfig[]
  │   ├── statuses: { topic[], learning[], default, completed[] }
  │   ├── types: TypeConfig[]
  │   └── workflows: WorkflowConfig[]
  └── advanced
      ├── tabs: TabConfig[]
      ├── fields: Record<string, string>
      ├── limits: Record<string, number>
      ├── theme: { defaultId, statusColors }
      └── templates: { weeklyCategories[] }
```
