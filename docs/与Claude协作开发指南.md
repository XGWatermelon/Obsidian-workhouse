## 一、项目启动阶段

### 1.1 CLAUDE.md — 你和 Claude 的「合同」

项目根目录的 `CLAUDE.md` 是 Claude 每次对话都会读的文件。它不是给人看的 README，是给 Claude 看的指令。

**本项目的 CLAUDE.md 是怎么写的**：

```markdown
# CLAUDE.md

## 项目概述
Obsidian 工作台插件 — 基于 TypeScript 开发的 Obsidian 插件。

## 构建与开发命令
npm run dev          # 开发模式
npm run build        # 生产构建

## 架构说明
main.ts — 插件入口
manifest.json — 插件元数据
...

## 编码准则 (Karpathy 原则)
- 最少代码解决问题。不做投机性设计。
- 只动必须动的。不顺手"改进"周围的代码。
- 遵循现有风格，即使你会写得不同。

## 修改记录规范
每次修改代码后，必须更新修改记录文档。
```

**关键点**：
- 编码准则选了 Karpathy 原则 — 因为老大的风格是「简洁优先，外科手术式改动」
- 修改记录规范是强制的 — 每次改动必须记录，这是老大 care 的点
- 不写教程 — Claude 已经会写 TypeScript，不需要教它怎么写

**踩过的坑**：

> **坑 1：CLAUDE.md 太长**
> 一开始写了很多架构细节，后来发现 Claude 每次对话都读它，太长会浪费上下文。精简到只写「约束」和「命令」。

> **坑 2：编码准则写了但没 enforce**
> 早期 Claude 有时会过度设计（比如引入不必要的抽象），后来在 CLAUDE.md 里加了「不加没被要求的功能」「只用一次的代码不抽象」，效果好了很多。

### 1.2 Memory 系统 — 跨会话的「长期记忆」

Claude 的 memory 系统存放在 `~/.claude/projects/<project>/memory/` 下，每个 memory 是一个 markdown 文件。

**本项目实际存了 6 个 memory**：

| Memory | 类型 | 内容 |
|--------|------|------|
| `user_preferences.md` | feedback | 不喜欢英文 spinner text |
| `project_claude_relay.md` | project | Obsidian ↔ VSCode 跨实例通信方案 |
| `reference_api_provider.md` | reference | 使用 mimo-v2.5-pro 代理，token 去代理商后台查 |
| `user_sap_skill_testing.md` | user | 正在测试 SAP HANA skills |
| `feedback_sap_sqlscript_hook.md` | feedback | sap-sqlscript 插件的 hook 会阻断所有对话 |
| `project_installed_plugins.md` | project | 安装了 context7 和 code-review 插件 |

**Memory 的实际作用**：

> **场景 1：跨会话记住用户偏好**
> 老大在一次对话中说「不喜欢英文 spinner text」，Claude 存了 memory。之后的新对话中，Claude 不会再问「要不要改成英文」，而是直接知道用户偏好。

> **场景 2：记住外部资源**
> 老大用的是 mimo-v2.5-pro 代理商，token 用量要去代理商后台查。存了 memory 后，新对话中 Claude 不会建议「去 Anthropic 控制台查」。

> **场景 3：记录技术方案**
> 跨实例 Claude 通信的方案（文件系统中转 + CLI 脚本）存了 memory，下次对话不需要重新分析。

**应该存什么，不应该存什么**：

| 应该存 | 不应该存 |
|--------|---------|
| 用户角色和偏好 | 代码结构（Claude 可以自己读） |
| 重要的设计决策和原因 | Git 历史（Claude 可以自己查） |
| 外部资源位置 | 临时任务状态 |
| 已确认的技术方案 | 已完成的工作细节 |
| 踩过的坑（非代码层面） | 代码层面的 bug 修复 |

**踩过的坑**：

> **坑 3：Memory 存太多会占上下文**
> Memory 文件会加载到 Claude 的上下文中。如果存了太多，会压缩留给实际工作的空间。6 个 memory 刚好，再多就该精简了。

> **坑 4：Memory 可能过时**
> `user_sap_skill_testing.md` 存的是「正在测试 SAP HANA skills」，但如果用户已经测完了，这个 memory 就过时了。需要定期清理。

### 1.3 Superpowers 技能 — Claude 的「工具箱」

本项目通过 `.claude/settings.json` 局部启用了 superpowers 插件，提供了 12 个技能：

**实际用过的技能**：
- `/brainstorming` — 讨论跨实例通信方案时
- `/writing-plans` — 编写插件实施计划时（14 个 Task）
- `/requesting-code-review` — 设计审查时（产出 3 份审查报告）
- `/verification-before-completion` — 每轮修复完成后

**没用过但应该用的**：
- `/systematic-debugging` — 热力图时区问题如果用这个技能，可能更快定位
- `/test-driven-development` — 如果先写测试再写代码，第一轮的 11 个问题可能不会出现

**避坑**：

> **坑 5：技能是建议，不是强制**
> Claude 有时会忽略技能流程，直接写代码。如果发现 Claude 跳过了某个步骤，可以明确要求：「先用 /brainstorming 讨论方案再动手」。

## 二、需求沟通阶段

### 2.1 需求的粒度

**本项目的实际需求演变**：

```
初始需求（5/27）：
  "做一个 Obsidian 工作台插件"
  → Claude 生成了 14 个 commit 的初始代码

第一轮反馈（5/28）：
  "插件只显示在侧边栏，需要在主页面打开"
  "今日要事一直显示加载中"
  "热力图月份和格子对不上"
  → 11 个具体问题，Claude 逐个修复

后续迭代：
  "选题改名为事项"
  "日记改名为日计划"
  "配置页面需要分层"
  "领域配置需要颜色选择器"
  → 每轮 3-23 个具体改动
```

**规律**：
- 初始需求可以模糊（「做一个工作台插件」），Claude 会按自己的理解实现
- 反馈必须具体（「热力图月份和格子对不上」而不是「热力图有问题」）
- 每轮反馈 3-5 个点最合适，太多 Claude 会遗漏

### 2.2 好反馈 vs 差反馈

**好反馈（本项目的真实例子）**：

```
"热力图月份和格子对不上。月份标签和热力图在不同容器，CSS Grid 无法对齐。"
→ Claude 知道是 CSS Grid 的问题，直接重写为 flex 布局

"'今日待办/本周待办'显示'...'没有实际数据。没有实现任务计数逻辑。"
→ Claude 知道是逻辑缺失，补全了计数函数

"创建笔记点击'创建'无反应。事项/目录不存在，仓库实际是选题/。"
→ Claude 知道是路径不匹配，重命名了仓库目录
```

**差反馈**：

```
"热力图有问题" → Claude 需要猜测是什么问题，可能猜错
"不好用" → Claude 完全不知道从哪改
"帮我改一下" → 改什么？
```

**避坑**：

> **坑 6：不要让 Claude 猜**
> Claude 不会读心术。「热力图有问题」和「热力图月份标签和格子对不上，应该是 CSS Grid 对齐问题」，后者 Claude 能一次修对，前者可能要猜 3 次。

### 2.3 设计讨论的节奏

**本项目的设计讨论案例**：

**节点文件夹位置的讨论**：

```
老大：节点应该放在哪里？

Claude 分析了 3 个选项：
  A. 工作台/节点/（默认推荐）
  B. 与工作台平级的节点/
  C. 用户自定义

老大：A 可以，但要支持 B（通过配置切换）

Claude 实现：name 包含 "/" 则为绝对路径，否则拼接 root
```

**核心设计原则的确立**：

```
老大：插件不应该篡改用户的文档结构

Claude：明确了边界划分
  - 插件只在工作台/下生成
  - 可以读取整个库
  - 不做移动/重命名
  - 不创建外部文件夹

→ 写入设计书 v1.1，成为项目的核心原则
```

**避坑**：

> **坑 7：不要让 Claude 自己做设计决策**
> Claude 会倾向于给出「最安全」的方案，但不一定是你想要的。比如节点位置，Claude 默认推荐放在工作台下，但如果你想要平级，需要明确说。

> **坑 8：设计决策要写进文档**
> 口头约定 Claude 会忘（新会话不记得）。重要的设计决策必须写进设计书或 CLAUDE.md。

## 三、代码实现阶段

### 3.1 初始代码的质量

**本项目的初始代码**：Claude 按实施计划生成了 14 个 commit，23 个文件，约 4,500 行。

**第一轮测试发现的 11 个问题**：

| 问题类型 | 数量 | 例子 |
|---------|------|------|
| 功能未实现（占位代码） | 3 | 工作流按钮只有 console.log、今日要事只显示"加载中" |
| CSS 布局问题 | 2 | tab 切换后布局乱、热力图对不齐 |
| 点击事件为空 | 2 | 今日待办/本周待办点击无效果 |
| 编译问题 | 1 | padStart 报错（tsconfig 配置不对） |
| 设计理解偏差 | 1 | 创建按钮都在选题/文件夹（设计如此，但用户不理解） |
| 代码清理 | 1 | 未使用的导入 |
| 路径问题 | 1 | 只显示在侧边栏 |

**规律**：
- Claude 生成的初始代码「框架是对的，细节是缺的」
- 占位代码（console.log、"加载中"）很多 — Claude 会先搭架子再填内容
- CSS 布局问题是最常见的 — Claude 对 CSS 的直觉不如 JS/TS
- 功能未实现的比功能错误的多 — Claude 会「承诺」但不「兑现」

**避坑**：

> **坑 9：初始代码不要直接用**
> Claude 生成的初始代码需要完整测试一遍。第一轮测试的 11 个问题，有 3 个是「功能根本没实现」。

> **坑 10：CSS 问题要自己看**
> Claude 对 CSS Grid/Flex 的计算经常出错。热力图对齐问题修了 3 轮才彻底解决。如果遇到 CSS 问题，最好自己打开 DevTools 检查。

### 3.2 迭代修复的节奏

**本项目的 10 轮迭代**：

| 轮次 | 日期 | 问题数 | 主要内容 |
|------|------|--------|---------|
| 1 | 05-28 | 11 | 基础功能修复（布局、加载、点击事件） |
| 2 | 05-28 | 3 | 选题→事项重命名、配置页面、说明书 |
| 3 | 05-28 | 5 | 热力图对齐、时区修复、错误处理 |
| 4 | 05-28 | 6 | 周计划两级结构、下拉框、日期选择器 |
| 5 | 05-28 | 8 | 热力图重构、布局修复、周日联动 |
| 6 | 05-28 | 23 | 大重构：节点/文件夹分离、主题系统、领域配置 |
| 7 | 05-29 | 4 | 常用入口改造、路径硬编码修复 |
| 8 | 05-29 | — | 硬编码审查（60+ 处） |
| 9 | 05-29 | 15 | 全面配置化改造（4 个新文件、13 个源文件改造） |
| 10 | 05-29 | 8 | ID 自动生成、模块编辑、tag 大小写、颜色选择器 |

**节奏规律**：
- 前 3 轮是「能用」阶段 — 修复基础功能
- 中间 3 轮是「好用」阶段 — 改进体验
- 后 4 轮是「通用」阶段 — 配置化改造

**每轮的理想模式**：

```
老大测试 → 发现问题 → 反馈给 Claude → Claude 修复 → 老大再测
```

**避坑**：

> **坑 11：不要一次改太多**
> 第六轮改了 23 个点，结果引入了新问题（路径不匹配、配置系统冲突）。后来第九轮做配置化改造时，专门拆成了 15 个子任务，每个都可以独立验证。

> **坑 12：修完一个测一个**
> Claude 有时会一口气改 5 个文件，然后说「改好了」。但如果其中一个文件改错了，其他 4 个的改动可能也需要回退。建议让 Claude 改完一个点就测试一个。

### 3.3 硬编码审查的教训

**本项目的硬编码审查（第八轮）**：

Claude 对整个项目做了硬编码审查，发现 60+ 处硬编码：

| 类别 | 数量 | 例子 |
|------|------|------|
| 文件夹路径 | 12 处 | `"事项/"`、`"SAP/模块/"` |
| 状态字符串 | 15 处 | `"进行中"`、`"待评估"` |
| 学习状态 | 5 处 | `"待阅读"`、`"已理解"` |
| 颜色值 | 8 处 | `"#FF6B6B"`、`"#77DD77"` |
| 数值常量 | 8 处 | `20`（最近文件数）、`500`（防抖延迟） |

**为什么会积累这么多硬编码**：

```
初始代码（5/27）：
  Claude 用硬编码快速实现功能 → 能用就行

第一轮~第七轮（5/28~5/29）：
  修 bug 时继续用硬编码 → 每次修一个路径

第八轮（5/29）：
  老大要求全面审查 → 发现 60+ 处硬编码

第九轮（5/29）：
  全面配置化改造 → 新增 4 个配置文件，改造 13 个源文件
```

**教训**：
- 硬编码是「温水煮青蛙」 — 每次加一点，最后积重难返
- 如果一开始就配置化，工作量反而更小
- 但「一开始」你可能还不知道需要配置什么 — 这是矛盾的

**避坑**：

> **坑 13：不要等积累了再清理**
> 如果发现第三次硬编码同一个东西（比如路径 `"事项/"`），就应该抽成配置了。不要等到 60+ 处。

> **坑 14：配置化改造要分步做**
> 第九轮一口气改了 13 个文件，结果第十轮又修了 8 个 bug。如果分 3 步做（先建配置系统 → 再逐个模块接入 → 最后测试），可能只需要 2 轮。

### 4.1 测试方法

配置功能：
  [ ] 修改配置后生效
  [ ] 主题切换正常
  [ ] 文件夹配置生效
  [ ] 状态/类型配置生效

### 4.2 常见 bug 模式

**本项目出现的 bug 模式**：

| 模式 | 频率 | 例子 |
|------|------|------|
| 路径不匹配 | 高 | 代码写 `"事项/"` 但实际文件夹是 `"选题/"` |
| 功能未实现 | 高 | console.log 占位、"加载中" |
| CSS 布局问题 | 高 | Grid 对不齐、Flex 不生效 |
| 时区问题 | 中 | toISOString() 返回 UTC |
| 大小写问题 | 中 | tag "agents" vs 配置 "AGENTS" |
| 空值处理 | 低 | 文件不存在时 crash |
| 编译问题 | 低 | tsconfig 配置不对 |

**避坑**：

> **坑 15：路径问题是最常见的 bug**
> 每次改文件夹结构后，都要 grep 一遍所有硬编码的路径。本项目从「选题」改名「事项」时，漏改了仓库目录，导致第三轮又要修。

> **坑 16：控制台是你的眼睛**
> 很多 bug 不会在界面上报错，但会在控制台打印。养成习惯：测试时打开 DevTools。

### 4.3 属性下拉框的测试教训

**本项目的属性下拉框（PropertyService）**：

代码写好了，注册了 Obsidian 的 types.json，理论上应该能用。但：
- 没有在实际 Obsidian 中测试过
- 不确定 Obsidian 的 select 属性是否支持动态更新
- 不确定编辑器补全是否在所有模式下都生效

**状态**：已实现未测试，在文档中标注。

**教训**：

> **坑 17：「写好了」不等于「能用」**
> Claude 说「写好了」只是说代码没有语法错误。实际能不能用，要在 Obsidian 里测。属性下拉框就是典型的「代码对但没验证」。

## 五、文档管理阶段

### 5.1 文档膨胀的过程

**本项目的文档演变**：

```
5/27：2 份文档（设计文档、整改方案）
5/28：+5 份（说明书、架构文档、逾期设计、通用化指导、测试记录）
5/29：+8 份（新用户指南、工作流程详解、硬编码指南、设计报告×3、整改方案×2）
      → 共 19 份文档，内容大量重叠
5/30：整理 → 19 份归档到 archive/，5 份新文档取代
```

**为什么会膨胀到 19 份**：

| 原因 | 例子 |
|------|------|
| 每次讨论都生成新文档 | 讨论通用化 → 通用化指导文档；讨论硬编码 → hardcode-upgrade-guide |
| 版本迭代不覆盖旧文档 | 设计报告 v1.0、v1.0.3、v1.1 三份并存 |
| 同一主题不同角度 | 上手指南、说明书、工作流程详解 内容重叠 70%+ |
| 一次性文档没清理 | 笔记库整改方案（已执行完毕）还在 docs/ 里 |

**避坑**：

> **坑 18：文档不要超过 5 份**
> 如果发现 docs/ 下超过 5 份文档，大概率有内容重叠。合并同类项。

> **坑 19：旧文档及时归档**
> 执行完毕的文档（如整改方案）应该立即归档。不要让它和活跃文档混在一起。

> **坑 20：让 Claude 写文档要审核**
> Claude 写的文档内容丰富但容易冗长。老大的做法是对的：让 Claude 写初稿，自己审核后精简。

### 5.2 文档整理的方法

**本项目的整理过程**：

```
1. 扫描所有文档 → 列出 19 份
2. 分类：哪些是活跃的，哪些是历史的
3. 归档历史文档 → docs/archive/
4. 合并活跃文档 → 5 份新文档
5. 更新 README 反映新结构
6. 更新 CLAUDE.md 的文档引用
```

**5 份新文档的分工**：

| 文档 | 面向 | 内容来源 |
|------|------|---------|
| 插件说明书 | 用户 | 说明书 + 上手指南 + 工作流程详解 |
| 架构文档 | 开发者 | 架构文档 + 通用化指导 + hardcode guide |
| 设计书 | 评审 | 设计文档 + 设计报告×3 + 逾期设计 |
| 开发指南 | 开发者 | GIT-WORKFLOW + 修改记录规则 |
| 协作指南 | 所有人 | 本指南 |

**避坑**：

> **坑 21：文档整理要在项目稳定后做**
> 如果在迭代中期整理文档，新改动会马上让整理结果过时。本项目在第十轮测试后才整理，此时代码基本稳定。

## 六、跨会话协作

### 6.1 上下文丢失的问题

**本项目遇到的上下文丢失**：

```
会话 1（5/27）：
  讨论设计 → 生成实施计划 → 生成初始代码
  → 上下文满了，自动压缩

会话 2（5/28）：
  老大继续测试 → Claude 不记得会话 1 的设计决策
  → 需要重新解释「选题」是什么

会话 3（5/29）：
  老大要求审查代码 → Claude 不记得之前改过什么
  → 需要重新读 git log
```

**解决方法**：

| 方法 | 适用场景 | 本项目的使用 |
|------|---------|-------------|
| CLAUDE.md | 持久化约束和命令 | 编码准则、修改记录规范 |
| Memory | 跨会话记忆 | 用户偏好、技术方案、外部资源 |
| 文档 | 项目持久化 | 设计书、测试记录、CHANGELOG |
| Git | 代码历史 | 7 次 commit，每次有清晰的 message |
| `/compact` | 压缩上下文 | 长对话后使用 |

**避坑**：

> **坑 22：重要决策不要只在对话里说**
> 对话会被压缩甚至丢失。设计决策写进设计书，技术方案写进 CLAUDE.md 或 memory。

> **坑 23：新会话先给上下文**
> 新会话开始时，简要说明「上次做到哪了」「这次要做什么」。不要假设 Claude 记得。

### 6.2 跨实例 Claude 通信

**本项目的尝试**：

老大想实现 Obsidian Claude ↔ VSCode Claude 之间的任务传递：
- Obsidian 中用 Claudian 插件测试插件
- VSCode 中用 Claude Code 开发插件
- 希望一个完成后自动通知另一个

**分析了 4 种方案**：

| 方案 | 可行性 | 问题 |
|------|--------|------|
| 文件系统中转 | 可行 | 需要轮询，延迟高 |
| MCP Server | 理论可行 | 需要两边都支持 MCP |
| 本地 HTTP 服务 | 可行 | 需要额外维护一个服务 |
| Claude API | 可行 | 需要 API key，费用高 |

**最终结论**：没有完美方案。最实际的是文件系统中转 + CLI 脚本。

**教训**：

> **坑 24：跨实例通信目前没有好方案**
> Claude 的各个实例（CLI、VSCode 扩展、Obsidian 插件）之间没有原生的通信机制。如果需要传递工作结果，最简单的方法是写到文件里，另一个实例读文件。

### 6.3 Claudian 插件的排查

**本项目遇到的 Claudian bug**：

```
问题：Obsidian 中的 Claudian 插件报错
  "messages[1].role must be either 'user' or 'assistant', but got 'system'"

排查过程：
  1. 对比两个 Obsidian vault 的插件版本
  2. 发现 working vault 用 v2.0.17（3,901,688 bytes）
  3. broken vault 用 v2.0.18（3,809,310 bytes）
  4. v2.0.18 有 bug，v2.0.17 正常

解决：把 v2.0.17 的文件复制到 broken vault
```

**教训**：

> **坑 25：第三方插件版本要锁定**
> 插件自动更新可能引入 bug。发现问题时，先检查版本变化。

> **坑 26：文件大小可以作为版本指纹**
> 同一个插件的不同版本，文件大小不同。如果不确定版本，比较文件大小是一个快速方法。

> 「fix: 修复路径硬编码问题，添加学习状态可点击功能」比「fix: bug fix」好 100 倍。

## 八、效率技巧

### 8.1 并行任务

Claude 可以同时处理多个独立任务。本项目的实际用法：

```
老大：帮我同时做三件事：
  1. 读一下 src/services/PlanService.ts 给我摘要
  2. 检查 main.ts 有没有问题
  3. 更新 CHANGELOG.md

Claude：同时启动 3 个 Agent，各自完成任务
```

### 8.2 `/compact` 的使用

长对话后使用 `/compact` 压缩上下文。本项目的实际效果：

```
压缩前：上下文占用 80%+
压缩后：上下文占用 30% 以下
保留：关键决策、文件路径、当前任务
丢失：中间讨论细节、已修复的 bug 详情
```

**什么时候用**：
- 对话超过 30 轮
- Claude 开始「忘记」前面说过的话
- 准备切换到新的话题

**什么时候不用**：
- 正在讨论一个重要设计决策
- 正在调试一个复杂 bug
- 需要 Claude 回忆之前的对话细节

### 8.3 代码审查的方式

本项目没有用正式的 PR 流程，但用了两种审查方式：

**方式 1：对话中审查**
```
老大：帮我审查一下 PlanService.ts 的代码

Claude：（读代码，列出 5 个潜在问题）
```

**方式 2：Superpowers 审查**
```
老大：/requesting-code-review

Claude：（按 code-review 技能的 9 个角度审查，产出报告）
```

**方式 3：设计审查**
```
老大：写一份设计审查报告

Claude：（分析架构、代码组织、设计模式，产出 3 份审查报告）
  - v1.0: 发现 6 个设计问题
  - v1.0.3: 新增节点设计分析
  - v1.1: 确立核心设计理念
```

## 十一、总结

### 黄金法则

1. **你做决策，Claude 做执行**
2. **需求要具体，反馈要清晰**
3. **小步快跑，每轮 3-5 个点**
4. **改完就测，测完就记**
5. **文档要精简，旧的要归档**
6. **重要决策写进文档，不要只在对话里说**
7. **CSS 问题自己看，不要完全信任 Claude**

### 效率公式

```
效率 = 清晰的需求 × 及时的反馈 × 严格的测试 × 持久化的记忆
```

### 最后

这份指南来自一个真实项目的 10 轮迭代、50+ 个 bug 修复、19 份文档的整理过程。每一个「坑」都是真实踩过的，每一个「解决」都是验证有效的。

你的项目可能不同，但这些模式会重复出现。记住：**Claude 是工具，你是主人。**
